Angular + PocketBase + Gotenberg

Documentación técnica de VerificarIT

VerificarIT es una PWA Angular para gestionar inspecciones vehiculares: autenticación, creación y edición de inspecciones, carga de fotografías, firmas digitales, consulta por placa, realtime con PocketBase y generación de reportes PDF mediante Gotenberg.

FrontendAngular 21 PWA
BackendPocketBase en EC2
PDFGotenberg en Dokploy
DatosSDK PocketBase JS

Arquitectura

La aplicación es un cliente Angular standalone. No existe un backend Node intermedio en el repositorio: Angular consume PocketBase directamente desde el navegador y envía archivos a Gotenberg para la conversión a PDF. 

Usuario navegador/PWA
  ├─ Angular 21 standalone + Service Worker
  │   ├─ AuthService: sesión PocketBase en cookie pb_auth
  │   ├─ InspectionService: CRUD inspections, images, secuencias
  │   ├─ RealtimeInspectionsService: subscribe('*') + localStorage
  │   └─ ExcelExportService + GotenbergService: XLSX -> PDF
  ├─ PocketBase EC2: https://db.buckapi.site:8095
  │   ├─ users
  │   ├─ inspections
  │   ├─ images
  │   └─ secuencias
  └─ Gotenberg Dokploy: https://gotenberg.buckapi.online
      ├─ /forms/libreoffice/convert
      └─ /forms/chromium/convert/html
UIComponentes standalone, formularios reactivos, SweetAlert2, Flatpickr, Signature Pad.
PersistenciaPocketBase SDK con HTTPS directo desde el cliente.
ArchivosImágenes en colección `images`, URLs vía `/api/files`.
ReportesPlantilla XLSX en assets, ExcelJS y conversión LibreOffice.

Realtime y caché

RealtimeInspectionsService mantiene una lista observable de inspecciones, eventos realtime y errores. La colección suscrita es inspections con acción comodín *. La caché local usa localStorage con clave inspections_cache, TTL de 5 minutos y sanitización de firmas para no inflar los listados.

  • Al crear, actualizar o eliminar se invalida caché para evitar lecturas antiguas.
  • El servicio pausa suscripciones si la sesión se queda sin token.
  • Los listados limitan campos con `INSPECTION_LIST_FIELDS` y excluyen firmas.

Flujo de generación de PDF

El flujo principal está en Detail.imprimirInspeccion(), ExcelExportService.exportarDatosConductorComoPdfConImagenes() y GotenbergService.convertXlsxToPdf().

Validación de estadoSi `estado` es `borrador`, se muestra una alerta con campos pendientes y se detiene la exportación.
Recolección de datosEl componente `Detail` arma un objeto plano desde `inspectionForm`, incluyendo conductor, vehículo, documentos, checklist, llantas, firmas y observaciones.
Imágenes existentesSe usa `this.inspectionImages`, que contiene URLs de archivos PocketBase ya resueltas.
Carga de plantilla`ExcelExportService` carga la plantilla XLSX desde `public/assets/templates` y busca la hoja `FIRST_PAGE`.
Procesamiento XLSXExcelJS escribe valores, radios, imágenes y firmas preservando estilos de la plantilla.
Blob en memoriaEl workbook se serializa como Blob `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`.
Conversión Gotenberg`GotenbergService` envía multipart/form-data con campo `files=inspeccion.xlsx`, `Accept: application/pdf` y Basic Auth.
DescargaEl PDF Blob recibido se descarga con nombre `Inspeccion_{placa}_{fecha}_CON_SECOND_PAGE.pdf`.
FormData Gotenberg:
files = inspeccion.xlsx

Headers:
Authorization: Basic ...
Accept: application/pdf

Endpoint:
https://gotenberg.buckapi.online/forms/libreoffice/convert
Existe también `convertHtmlToPdf()` contra Chromium con opciones como `printBackground`, `waitForSelector`, `landscape`, tamaños de papel y márgenes. No es el camino principal de `imprimirInspeccion()`.

Variables de entorno

El proyecto actual no define `src/environments/environment.ts`; las URLs y credenciales de Gotenberg están hardcodeadas en servicios/proxy. Para despliegue profesional deben moverse a configuración por ambiente y secretos de plataforma.

Recomendación inmediata: rotar credenciales expuestas, retirarlas del repositorio y poner Gotenberg detrás de un proxy controlado por dominio/CORS o una función backend que firme la solicitud.

Guía de despliegue

Frontend en AWS Amplify

  1. Conectar el repositorio y seleccionar la rama productiva.
  2. Configurar build: npm ci y npm run build.
  3. Publicar artefacto desde dist/verificar-app/browser.
  4. Configurar redirección SPA: todas las rutas a /index.html con 200.
  5. Permitir el dominio Amplify en reglas/CORS de PocketBase y Gotenberg.
  6. Validar Service Worker, manifest, iconos PWA y caché después del primer despliegue.
version: 1
frontend:
  phases:
    preBuild:
      commands:
        - npm ci
    build:
      commands:
        - npm run build
  artifacts:
    baseDirectory: dist/verificar-app/browser
    files:
      - '**/*'
  cache:
    paths:
      - node_modules/**/*

PocketBase en EC2

  1. Instalar PocketBase compatible con el SDK `pocketbase@0.26.6`.
  2. Ejecutarlo como servicio systemd con volumen persistente para pb_data.
  3. Exponerlo detrás de HTTPS con Nginx/Caddy o balanceador.
  4. Importar/crear colecciones `users`, `inspections`, `images`, `secuencias`.
  5. Configurar reglas de acceso y CORS para el dominio Amplify.
  6. Programar backups de pb_data y probar restauración.
[Unit]
Description=PocketBase
After=network.target

[Service]
WorkingDirectory=/opt/pocketbase
ExecStart=/opt/pocketbase/pocketbase serve --http=127.0.0.1:8090
Restart=always
User=pocketbase

[Install]
WantedBy=multi-user.target

Gotenberg en Dokploy

  1. Crear app Docker en Dokploy con imagen gotenberg/gotenberg:8.
  2. Asignar dominio HTTPS, por ejemplo gotenberg.example.com.
  3. Activar protección por Basic Auth en proxy/Dokploy o delante de Traefik.
  4. Permitir payloads suficientes para XLSX con imágenes.
  5. Probar /forms/libreoffice/convert con un XLSX mínimo.
  6. Configurar CORS si el navegador llama directo al servicio.
services:
  gotenberg:
    image: gotenberg/gotenberg:8
    restart: unless-stopped
    ports:
      - "3000:3000"

Troubleshooting

SíntomaCausa probableAcción
Login falla con 401Credenciales inválidas, cookie corrupta o reglas PocketBase.Limpiar `pb_auth`, verificar usuario en `users`, revisar HTTPS y hora del cliente.
Listados muy lentosCampos grandes como firmas o imágenes en consultas.Usar `INSPECTION_LIST_FIELDS`, no traer blobs/base64 en listados, agregar índices por placa/estado/created.
Realtime no actualizaWebSocket/SSE bloqueado, token expirado o CORS.Revisar `/api/realtime`, proxy HTTPS, logs de PocketBase y renovación de sesión.
PDF no se generaGotenberg inaccesible, Basic Auth incorrecto, endpoint equivocado.Probar `POST /forms/libreoffice/convert`, revisar CORS, tamaño de request y credenciales.
Imágenes no aparecen en PDFURLs PocketBase sin permisos, CORS de archivos o formato no soportado.Validar lectura pública/autenticada de `/api/files`, convertir imagen descargada a formato compatible con ExcelJS.
Plantilla fallaNo existe hoja `FIRST_PAGE` o asset no publicado.Verificar `public/assets/templates`, nombre de hoja y ruta en build Amplify.
Certificado duplicadoIncremento de `secuencias` sin transacción.Implementar bloqueo/endpoint backend o regla server-side para incremento atómico.
Rutas dan 404 en AmplifyFalta rewrite SPA.Redirigir `/<*>` a `/index.html` con status 200.
PWA no actualizaService Worker cacheando build anterior.Forzar refresh, revisar `ngsw-config.json`, confirmar hash de build y headers de cache.

Seguridad y deuda técnica

  • Las credenciales de Gotenberg están en el frontend; cualquier usuario puede extraerlas del bundle. Deben rotarse y moverse fuera del cliente.
  • El token PocketBase se exporta a cookie con httpOnly: false, una limitación normal desde frontend pero sensible ante XSS.
  • Las reglas de PocketBase no están versionadas; sin ellas no se puede auditar autorización real.
  • Evitar hardcodear collection IDs como `images`; exponerlos desde configuración o resolverlos por metadata.
  • Evitar archivos temporales o credenciales accidentales en el repositorio antes de publicar.
